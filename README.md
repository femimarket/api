# Rust FFI for Femi API

A cross-platform Rust library providing a C-compatible Foreign Function Interface (FFI) for the Femi AI API. It exposes image generation, video synthesis, audio transcription, and LLM chat endpoints to native consumers (Swift/iOS/macOS, Kotlin/Android, and WebAssembly/Kotlin Multiplatform).

It also includes a `ProjectService` module that manages local file storage and XMP metadata embedding, abstracting away platform-specific file I/O (Sandboxed Documents on Apple, `filesDir` on Android, OPFS on Web).

## Architecture

The project is split into two main layers:

1.  **Rust Core (`Rust/`)**: The source of truth. Contains the FFI bindings, API client logic, and platform-specific backends for file storage.
2.  **Consumers**:
    *   **Swift (`Sources/Api/`)**: Thin wrappers around the C ABI for iOS/macOS.
    *   **Kotlin/Android (`Kmp/api/src/androidMain/`)**: JNI bindings for native libraries.
    *   **Kotlin/Web (`Kmp/api/src/webMain/`)**: WASM bindings with embedded JS/WASM assets.

### Key Directories

*   `Rust/src/api/`: Individual modules for each AI endpoint (e.g., `flux2_pro.rs`, `qwen3_asr_flash.rs`). Each module contains:
    *   `core_*`: Pure async Rust logic (HTTP requests, JSON parsing).
    *   `native`: C-ABI wrappers for Apple/Android (handles string marshaling, cancellation flags, memory allocation).
    *   `wasm`: `wasm_bindgen` exports for Web.
*   `Rust/src/project_service/`: File management and XMP metadata.
    *   `shared/xmpkit_body.rs`: Platform-agnostic XMP embedding logic (used by Android and WASM).
    *   `apple.rs`: Uses `xmp_toolkit` for smart XMP handling on Apple.
    *   `android.rs`: JNI wrappers for Android file I/O.
    *   `wasm.rs`: Web API (OPFS) wrappers for browser storage.
*   `RustFFI.xcframework/`: Pre-built Apple framework (generated by build script).
*   `Kmp/api/src/androidMain/jniLibs/`: Pre-built Android `.so` libraries (generated by build script).
*   `Kmp/api/src/webMain/kotlin/market/femi/api/RustFfiBundle.kt`: Base64-encoded WASM glue code (generated by build script).

## Build & Install

### Prerequisites

*   **Rust**: `rustup` installed.
*   **Android NDK**: Required for Android targets. Ensure `ANDROID_NDK_HOME` or `NDK_HOME` is set, or that the NDK is installed via Android Studio and referenced in `Rust/.cargo/config.toml`.
*   **Xcode**: Required for Apple targets (for `xcodebuild`).
*   **Node.js**: Required for the WebAssembly inlining step.
*   **wasm-bindgen-cli**: Installed automatically by the build script if missing.

### Building Everything

Run the top-level build script from the repository root:

```bash
./build-rust.sh
```

This script performs the following steps:

1.  **Installs Targets**: Adds `aarch64-apple-ios`, `aarch64-apple-ios-sim`, `aarch64-apple-darwin`, `wasm32-unknown-unknown`, and Android targets (`arm64-v8a`, `armeabi-v7a`, `x86_64`, `x86`) via `rustup`.
2.  **Apple Build**: Compiles the static library for Apple targets and bundles them into `RustFFI.xcframework`.
3.  **Android Build**: Compiles the shared library for Android ABIs and copies them to `Kmp/api/src/androidMain/jniLibs/<abi>/librust_ffi.so`.
4.  **WebAssembly Build**:
    *   Compiles `wasm32-unknown-unknown`.
    *   Runs `wasm-bindgen` to generate JS glue code.
    *   **Inlines the `.wasm` binary** into the JS glue code as a Base64 string. This creates a single self-contained `pkg/rust_ffi.js` that does not require a separate `.wasm` file fetch.
    *   **Embeds the JS glue code** into a Kotlin constant `RustFfiBundle.kt` inside the KMP source tree.

### Manual Build (Development)

If you only need to rebuild the Rust crate (e.g., after changing logic):

```bash
cd Rust
cargo build --release --target aarch64-apple-ios
cargo build --release --target aarch64-apple-darwin
# ... other targets
```

## Usage

### Swift (iOS/macOS)

Import the `Api` package in your `Package.swift`:

```swift
.package(path: "../") // Or reference the local path
```

Use the `Api` enum for AI endpoints:

```swift
import Api

let image = await Api.flux2Pro(
    user: "my_user",
    password: "my_pass",
    prompt: "a futuristic city"
)
```

Use `ProjectService` for local file management:

```swift
import Api

// Save a file with metadata
ProjectService.saveFile(imageData, named: "city.png", prompt: "futuristic city", model: "flux2-pro")

// Retrieve metadata
let prompt = ProjectService.getPrompt("city.png")
let liked = ProjectService.getLike("city.png")
```

### Kotlin Multiplatform (Android/Web)

The KMP module exposes the FFI via `FemiApiJvm` (Android) and `RustFfi` (Web).

**Android:**
```kotlin
// Automatically loads librust_ffi.so
class FemiApiJvm {
    companion object { init { System.loadLibrary("rust_ffi") } }
    
    external fun rustFfiFlux2Pro(user: String, password: String, prompt: String, cancelFlag: Long): ByteArray
}

// Usage
val result = FemiApiJvm.rustFfiFlux2Pro(user, pass, prompt, 0)
```

**Web (Kotlin/Wasm):**
The WASM module is embedded in `RustFfiBundle.kt`. Initialize it once:

```kotlin
import market.femi.api.RustFfi

// Initialize the embedded WASM module
RustFfi.init()

// Call endpoints
val result = RustFfi.wasmFlux2Pro(user, pass, prompt)
```

### C/C++

Link against `RustFFI.xcframework` (Apple) or `librust_ffi.so` (Android). Include `Rust/include/RustFFI/RustFFI.h`.

```c
#include "RustFFI.h"

size_t len = 0;
uint8_t* result = rust_ffi_flux2_pro("user", "pass", "prompt", NULL, &len);
// ... use result ...
// Note: The caller is responsible for freeing the returned pointer if it is not null.
// In Rust, this is done via Box::into_raw, so you must use free() or equivalent.
```

## API Endpoints

The following endpoints are exposed via FFI. All return `uint8_t*` (bytes) and `size_t` (length).

| Function | Description | Inputs |
| :--- | :--- | :--- |
| `rust_ffi_z_image_turbo` | Generate image via ZImageTurbo | `user`, `password`, `prompt` |
| `rust_ffi_flux2_pro` | Generate image via Flux2 Pro | `user`, `password`, `prompt` |
| `rust_ffi_flux2_dev_i2i` | Image-to-Image via Flux2 Dev | `user`, `password`, `image_b64`, `prompt` |
| `rust_ffi_flux2_klein_i2i` | Image-to-Image via Flux2 Klein | `user`, `password`, `image_b64`, `image2_b64`, `prompt` |
| `rust_ffi_nano_banana2` | Generate image via Nano Banana2 | `user`, `password`, `prompt` |
| `rust_ffi_ltx2_3a2v` | Audio-to-Video via LTX2 | `user`, `password`, `image_b64`, `audio_b64`, `prompt` |
| `rust_ffi_qwen3_asr_flash` | Audio transcription (ASR) | `user`, `password`, `audio_b64` |
| `rust_ffi_qwen3_6_35b_a3b` | LLM Chat (Qwen3) | `user`, `password`, `messages_json` |

### Cancellation

All API functions accept a `cancel_flag` pointer (`const uint8_t*`). If this pointer is non-null, the Rust side polls the value at that address. If the value becomes non-zero, the operation is cancelled and returns a fallback image/video/text.

### Memory Management

*   **Return Values**: Functions return a pointer to a heap-allocated buffer (`uint8_t*`). The length is written to the `out_len` pointer.
*   **Freeing**: The caller **must** free the returned buffer. In Swift/Kotlin, this is handled by the wrapper (using `Data(bytesNoCopy: ... deallocator: .free)` or JNI byte array conversion). In C/C++, use `free()`.

## ProjectService (Local Storage & XMP)

The `ProjectService` module manages files in the app's Documents directory. It embeds XMP metadata (prompt, model, subjects) into images and videos.

### Apple
Uses `xmp_toolkit` to read/write XMP metadata using smart handlers. Files are stored in the app's `Documents/` directory.

### Android
Uses JNI to read/write files directly from `context.filesDir`. XMP metadata is embedded using a pure-Rust bytes-based implementation (`xmpkit_body`).

### Web (WASM)
Uses the Origin Private File System (OPFS). Files are stored in a flat namespace per origin. XMP metadata is embedded using `xmpkit_body`.

### Key Functions

*   `psxmp_save_file`: Save a file with optional prompt, model, and subject metadata.
*   `psxmp_save_audio`: Save an audio file (replaces any existing audio file).
*   `psxmp_like`: Set/unset a "like" rating on a file.
*   `psxmp_get_all_generations`: List all generated files.
*   `psxmp_get_prompt`, `psxmp_get_model`, `psxmp_get_subject`: Read metadata from a file.
*   `psxmp_set_character_cast`, `psxmp_get_character_cast`: In-memory state for character casting (not persisted to disk).
*   `psxmp_set_image_edit`, `psxmp_get_image_edit`: In-memory state for image editing context.

## Testing

### Rust Tests

Run Rust integration tests against the live API:

```bash
cd Rust
cargo test
```

Tests create a fresh user account per run (`funded-test-<uuid>`) and verify that the API returns real images/videos instead of fallback assets.

### Swift Tests

Run Swift tests using Xcode or `swift test`. Tests use the `Api` package and verify cancellation behavior and metadata embedding.

### Android Tests

Android tests are part of the KMP module and run on the Android emulator.

## Non-Obvious Conventions

1.  **Base64 Encoding**: Image and audio inputs are passed as Base64-encoded strings to the FFI. The Rust side decodes them for the HTTP request.
2.  **Fallback Assets**: If the API returns an error (e.g., 402 Payment Required, or network failure), the Rust FFI returns embedded fallback assets (`fallback.png`, `could-not-generate.mp4`, etc.) instead of null/empty. This ensures the UI always has something to display.
3.  **WASM Inlining**: The Web build process inlines the `.wasm` binary into the JS glue code as Base64. This eliminates the need for a separate `.wasm` file fetch, simplifying deployment for web apps.
4.  **Kotlin Web Embedding**: The JS glue code is further embedded into a Kotlin constant (`RustFfiBundle.kt`). This allows the WASM module to be shipped entirely within the KMP library, without requiring external npm packages or local file paths.
5.  **Cancellation**: Cancellation is cooperative. The Rust side polls a memory location. This works across FFI boundaries because the pointer is passed as an integer address.
